SAP Commerce Agent Architecture

System overview and LangGraph execution flow

Problem Statement Why This Exists

SAP Commerce Cloud is powerful but complex. Customers, support teams, and business users struggle with its steep learning curve, rigid UI, and multi-step workflows. This project explores whether an AI agent can make e-commerce interactions as simple as a conversation.

Challenge 1

Complex Multi-Step Checkout

Placing an order in SAP Commerce requires navigating through product search, cart management, delivery address, delivery mode, payment details, and order confirmation — each with its own API call, validation rules, and error handling. Users must understand the exact sequence; a wrong step means starting over.

Solution

Conversational Orchestration

A LangGraph agent handles the entire checkout flow through natural language. The user simply says "I want to buy a Canon EOS camera" and the agent orchestrates all 5+ API calls in the correct order, handling errors and retries automatically. The graph's state machine ensures steps are never skipped or misordered.

Challenge 2

API Integration Complexity

SAP Commerce OCC v2 API has 100+ endpoints with complex authentication (OAuth2), session management, CSRF tokens, and domain-specific conventions (user_id="current" vs "anonymous", cart codes vs GUIDs). Building and maintaining a custom frontend is expensive and rigid.

Solution

MCP Tool Abstraction

An MCP (Model Context Protocol) server wraps 18 SAP OCC endpoints as discoverable tools with clean schemas. The AI agent discovers and calls these tools dynamically. A server-side Token Vault keeps OAuth tokens secure — the LLM never sees raw credentials. New tools can be added without changing the agent.

Challenge 3

Resilience Under Load

Both external dependencies — SAP Commerce and Anthropic API — can fail. SAP may go down for maintenance, return SSL errors, or timeout. Anthropic may return 529 overloaded errors during peak usage. Without resilience patterns, one failure cascades into a broken user experience.

Solution

Circuit Breaker + Retry with Backoff

A circuit breaker tracks SAP failures and stops making calls when the service is down, returning a friendly message instead. Anthropic 529 errors trigger exponential backoff with jitter (2s, 4s, 8s, 16s). SSL errors produce structured, actionable log entries with specific fix instructions for the operations team.

The Big Picture

This project Tech:

  • LangGraph for stateful, multi-step agent workflows with human oversight
  • MCP (Model Context Protocol) for standardized, secure tool integration
  • Claude for natural language understanding and tool orchestration
  • SAP Commerce Cloud OCC v2 as the commerce backbone

We create a shopping experience where a user can go from "show me cameras under $500" to a placed order in a single conversational thread — with full security, observability, and human-in-the-loop controls at every critical step.

Request Flow: User sends a message via the Chat UI → api_server.py receives it at POST /chat → Security Layer checks for prompt injection, rate limits, and sanitizes input → LangGraph Agent builds the system prompt, trims context, and invokes Claude → Claude decides which tools to call → MCP Client opens an SSE connection to the MCP Server → MCP Server resolves the session token from the Token Vault and calls SAP Commerce OCC API → results flow back through the graph.

Login flow (dashed red) bypasses the LLM entirely — credentials go directly from the UI to /auth/login, are used once for OAuth, then discarded.

LangGraph execution cycle:
  • agent_node — Builds the system prompt with dynamic session state (auth, cart_id, session_id), trims the context window while preserving tool call/result pairs, and invokes Claude with bind_tools(ALL_TOOLS). Includes exponential backoff retry for Anthropic 529 overloaded errors.
  • route_after_agent — If Claude's response contains place_order, route to human_approval. If it has other tool calls, route to tools. If it's a plain text reply, route to sync.
  • human_approval_node — Uses langgraph.types.interrupt() to pause the graph. The API returns awaiting_approval: true to the frontend, which shows confirm/cancel buttons. When the user responds, the graph resumes via Command(resume=...).
  • tool_node_with_injection — Automatically injects the session_id / access_token from state into every tool call (Claude never handles raw tokens). Rejected tool calls from the approval step return a cancellation message instead of executing.
  • state_sync_node — Scans tool results and extracts state updates: cart_id, order_code, user_id, etc. Also feeds the circuit breaker (opens after repeated SAP failures).
  • route_after_sync — If the last message is a ToolMessage, loop back to agent_node so Claude can interpret the results. Otherwise, the turn is complete → END.

MCP Tool Call Flow Detail

Tool call lifecycle: Claude decides to call a tool (e.g., search_products(query="cameras")) → tool_injection automatically adds session_id from state (Claude never handles tokens) → MCP Client opens a fresh SSE connection and calls session.call_tool() → MCP Server resolves the session from the Token Vault, builds the SAP OCC API request with the real OAuth token → SAP Commerce returns the result → response flows back as a ToolMessage → the graph loops back to agent_node where Claude interprets the result and responds to the user.
